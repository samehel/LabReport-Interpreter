# LabReport Interpreter — Project Design Document

## Overview

A web-deployed Flutter application that allows users to upload or manually enter medical lab test results, receive plain-language summaries, track trends across visits, and get flagged alerts for abnormal values. The backend is a Python REST API with an ML pipeline for OCR extraction and NLP-based interpretation.

This is a master's-level demonstration project. The design prioritizes clarity, reasonable scope, and demonstrable ML features over production-grade scalability.

## 2. Python Backend

### 2.1 Framework

**FastAPI**

Reasons:
- Fastest development cycle for REST APIs in Python
- Auto-generated Swagger UI (`/docs`) — useful for demo and testing
- Native async support
- Easy integration with ML libraries (no ORM or framework lock-in)
- Pydantic models for automatic request/response validation

---

### 2.2 Architecture

**Pattern: Layered Monolith (suitable for master's scope)**

```
app/
├── main.py                  # FastAPI app init, router registration
├── config.py                # Settings, env vars (via pydantic-settings)
├── database.py              # DB connection (SQLAlchemy or Motor for Mongo)
│
├── routers/
│   ├── auth.py              # Register, login, token endpoints
│   ├── reports.py           # Upload, list, get, delete reports
│   ├── metrics.py           # Get trend data per metric
│   └── summary.py           # Trigger ML pipeline, return summary
│
├── models/
│   ├── user.py              # User DB model
│   ├── report.py            # Report DB model
│   └── metric.py            # Individual lab value model
│
├── schemas/
│   ├── auth.py              # Request/response Pydantic schemas
│   ├── report.py
│   └── metric.py
│
├── services/
│   ├── auth_service.py      # Registration, login logic, JWT handling
│   ├── report_service.py    # CRUD, file storage
│   └── ml_service.py        # OCR + NLP pipeline orchestration
│
└── ml/
    ├── ocr.py               # Extract structured data from PDF/image
    ├── classifier.py        # Normal/abnormal classification
    └── summarizer.py        # Plain-language generation
```

---

### 2.3 Database

**PostgreSQL** via SQLAlchemy (async with `asyncpg`)

Alternatively: **SQLite** for local demo simplicity (swap via config).

Tables:
- `users` — id, name, email, hashed_password, dob, created_at
- `reports` — id, user_id, uploaded_at, source (file/manual), notes, file_path
- `lab_values` — id, report_id, test_name, value, unit, ref_low, ref_high, status

---

### 2.4 ML Pipeline

#### Step 1 — OCR Extraction (`ml/ocr.py`)
- Library: `pytesseract` + `pdf2image` (for PDF input)
- Alternatively: `pdfplumber` for text-based PDFs (preferred when available — more accurate)
- Output: raw text or semi-structured key-value pairs

#### Step 2 — Value Parsing (`ml/ocr.py`)
- Library: `spaCy` or regex-based NLP to extract (test name, value, unit, range) tuples
- Rule-based extraction is acceptable for master's scope
- Fallback: if confidence low, flag for manual review

#### Step 3 — Abnormality Classification (`ml/classifier.py`)
- Rule-based: compare extracted value against reference range → Normal / Low / High / Critical
- ML enhancement: train a small sklearn `RandomForestClassifier` or `LogisticRegression` on a sample dataset of lab result patterns to classify contextual risk (e.g., combination of low hemoglobin + low MCV = likely iron deficiency)
- Dataset: use publicly available MIMIC-III derived or synthetic lab result datasets

#### Step 4 — Plain-Language Summary (`ml/summarizer.py`)
- Use `transformers` library with a small pre-trained model
- Recommended: `facebook/bart-large-cnn` (summarization) or a fine-tuned `GPT-2` on medical QA
- For demo scope: template-based generation is acceptable as fallback
- Input: structured JSON of lab values + statuses
- Output: 3–5 sentence plain-language paragraph

#### Step 5 — Correlation Detection
- Simple statistical check: `scipy.stats.pearsonr` between metric pairs across reports
- Flag correlated abnormal pairs with a brief pre-written clinical note

### 2.6 API Endpoints (Summary)

```
POST   /auth/register
POST   /auth/login              → returns JWT

GET    /reports                 → list user's reports
POST   /reports/upload          → upload PDF/image file
POST   /reports/manual          → submit manually entered values
GET    /reports/{id}            → full report with lab values + summary
DELETE /reports/{id}

GET    /metrics/{test_name}/trend   → values over time for charting

POST   /summary/{report_id}         → trigger ML pipeline (if not auto-run)
GET    /summary/{report_id}/pdf     → download generated PDF
```

### 2.7 Security

Scope is a master's demo — apply reasonable, not exhaustive, security.

| Concern | Implementation |
|---|---|
| Authentication | JWT Bearer tokens via `python-jose`, expiry: 24h |
| Password storage | `bcrypt` hashing via `passlib` |
| Input validation | Pydantic schemas on all endpoints |
| File upload safety | Validate MIME type and file size limit (≤10MB) |
| CORS | Restrict to Flutter web origin in production config |
| SQL injection | Prevented by SQLAlchemy ORM (no raw queries) |
| Rate limiting | `slowapi` middleware (basic, per-IP) |

No HTTPS configuration needed for local demo; add if deploying to a public URL.

### 2.8 Deployment (Web Demo + Mobile Access)

Flutter compiles to both a web build and native mobile builds from the same codebase. This project supports two access modes simultaneously:

#### Mode 1 — Browser (Desktop/Laptop Demo)
- Run `flutter build web` to produce a static web bundle
- Serve via Nginx or `python -m http.server` on a local or hosted machine
- Accessible from any browser at the served URL
- Ideal for live demonstration and evaluation

#### Mode 2 — Mobile Phone (Android/iOS)
- Run `flutter build apk` (Android) or `flutter build ios` (iOS, requires macOS + Xcode)
- Install the APK directly on an Android device via USB or file transfer — no Play Store required
- iOS requires either a developer certificate sideload or TestFlight — for demo purposes, Android APK is the simpler path
- The mobile app connects to the same FastAPI backend via its IP/URL; no backend changes needed

#### Backend (Shared by Both)
| Component | Tool |
|---|---|
| FastAPI backend | `uvicorn app.main:app --host 0.0.0.0 --port 8000` |
| Database | PostgreSQL local instance or SQLite for simplicity |
| ML models | Loaded at startup, cached in memory |
| Environment | `.env` file via `pydantic-settings` |

The backend is access-mode agnostic — both the browser app and the mobile app communicate with it over HTTP. If deployed to a public URL (e.g., a VPS or ngrok tunnel), both modes work without any reconfiguration beyond updating the base API URL in the Flutter config.

## 3. Scope Boundaries (Master's Level)

The following are intentionally out of scope to keep the project feasible:

- No real-time notifications
- No third-party EHR or hospital system integration
- No multi-language support
- No mobile app store builds
- No production-grade model training (use pre-trained + fine-tuned or rule-based ML)
- No admin dashboard
- No payment or subscription logic
